## 기획자 의견

### 추천 발표 주제 각도
"13F 데이터로 미국 기관투자자의 포지션을 시각화한다"로 주제를 좁힌다. Hood chain 연동은 발표용 Mock JSON 출력으로 대체해 "데이터 수집 → 시각화 → 온체인 전송 준비"라는 완결된 흐름을 보여준다.

### Day8 핵심 장면
로컬에 저장한 13F XML 파일을 업로드 버튼으로 선택하면, 종목명·보유수량·비중(%) 테이블이 즉시 화면에 렌더링되는 장면. 클릭 한 번으로 결과가 바뀌는 모습이 발표 하이라이트가 된다.

### 발표 가능 수준으로 줄이는 방법
원래 주제(자동 수집 + 온체인 포트폴리오 변환)는 2일 범위를 초과한다. "자동 수집"을 "수동 파일 업로드"로, "온체인 트랜잭션"을 "Mock 트랜잭션 JSON 미리보기"로 교체하면 Day7 디자인 + Day8 구현이 충분히 가능하다. 핵심 가치(13F 파싱·비중 시각화)는 그대로 유지된다.

## 구현자 의견

### Day7에서 할 것 (디자인)
- 화면 구성 정의: 상단 헤더(프로젝트 제목), 중앙 테이블(13F 보유 종목 목록), 하단 액션 버튼 영역
- 와이어프레임 작성: 종목명·보유수량·비중(%) 3개 컬럼 테이블 레이아웃 스케치
- HTML 골격 잡기: `index.html` 기본 구조(nav, main, section) 작성, CSS 변수 정의(색상, 폰트, 간격)
- 샘플 데이터 설계: 실제 API 없이 하드코딩된 JSON 픽스처로 화면이 정상 렌더링되는지 확인
- 모바일 반응형 기준선 설정: 텍스트 겹침 없는 최소 레이아웃 확인

### Day8에서 할 것 (구현)
- 13F XML 파일 업로드 파싱: `<input type="file">` + `FileReader` + `DOMParser`로 로컬 XML을 읽어 종목 배열로 변환
- 비중 계산 로직: 각 종목의 보유 수량을 전체 합계로 나눠 비중(%)을 계산하는 순수 함수 구현
- localStorage 저장·불러오기: 파싱 결과를 `localStorage.setItem`으로 저장, 재방문 시 `getItem`으로 복원
- 테이블 동적 렌더링: 파싱된 배열을 `<tbody>` DOM에 삽입하는 JS 함수 작성
- 배포 가능 상태 확인: 외부 서버 없이 `file://` 또는 GitHub Pages에서 정상 동작 검증

### 핵심 기능 3개 이하 (브라우저 단독 실행 가능한 것만)
1. **13F XML 로컬 파일 업로드 및 파싱** — FileReader + DOMParser로 브라우저에서 직접 처리, 서버 불필요
2. **종목별 보유 비중 계산 및 테이블 표시** — 순수 JS 연산, 외부 의존 없음
3. **localStorage 기반 세션 유지** — 새로고침 후에도 마지막 파싱 결과 복원 (선택)

### 외부 API 없이 가능한지 판단
**가능한 부분 (브라우저만으로 처리 가능):**
- 로컬에 다운로드한 13F XML 파일을 FileReader로 읽어 파싱하는 것은 완전히 브라우저 내에서 처리 가능
- 종목명·수량·비중 계산 및 테이블 렌더링은 순수 JS로 구현 가능
- localStorage로 데이터 저장·복원 가능

**불가능한 부분 (외부 API 없이는 불가):**
- SEC EDGAR에서 최신 13F 파일을 자동으로 가져오는 것 — CORS 제한으로 브라우저에서 직접 fetch로 SEC 서버에 접근 불가, 사용자가 수동으로 파일을 다운로드해 업로드하는 방식으로 대체 필요
- Hood chain 지갑 연동 및 온체인 트랜잭션 생성 — 브라우저 지갑 확장(MetaMask 등)이 필요하며, Hood chain RPC 엔드포인트 접근이 필수이므로 외부 네트워크 연결 없이는 불가
- 실시간 토큰 가격·시세 반영 — 외부 가격 API 없이는 구현 불가, Day7·8 범위에서 제외 권장

**결론:** Day7·8 범위는 "로컬 XML 업로드 → 파싱 → 테이블 표시 → localStorage 저장"으로 한정하고, Hood chain 연동은 데모용 모의(Mock) 트랜잭션 JSON 출력 수준으로 제한하는 것이 현실적이다.

## 검토자 의견

### 범위 위험 (너무 커서 2일 안에 끝낼 수 없는 부분)

- **Hood chain 지갑 연동**: 구현자도 지적했듯 MetaMask 등 브라우저 지갑 확장과 Hood chain RPC 엔드포인트 연결이 필요하다. 이를 Day8 하루 안에 구현하면 지갑 설정·테스트넷 환경 구성·트랜잭션 서명 흐름 전체를 다뤄야 하므로 실질적으로 시간 초과 위험이 높다. 기획자 의견의 "Mock 트랜잭션 JSON 출력"이 현실적인 상한선이다.
- **SEC EDGAR 자동 수집**: CORS 제약으로 브라우저에서 직접 SEC 서버에 fetch 요청을 보낼 수 없다. "자동 수집"이라는 표현이 주제에 포함되어 있으나, Day7·8 범위에서는 사용자가 수동으로 다운로드한 XML 파일을 업로드하는 방식으로 제한해야 한다. 자동화를 넣으려면 CORS 프록시 또는 백엔드 서버가 필요해 범위가 즉시 두 배 이상 커진다.
- **RWA 포트폴리오 변환·관리**: 주제 문장에 포함된 "변환·관리" 개념은 온체인 트랜잭션 전송, 토큰화 계약 호출, 포트폴리오 상태 추적까지 포함할 수 있다. 이 전체를 Day8에 넣으면 완성 불가능하다. Mock JSON 출력 수준으로 명시적으로 축소해야 한다.

### API 키·외부 토큰 위험

- **SEC EDGAR API**: SEC의 EDGAR Full-Text Search API나 데이터 다운로드 엔드포인트는 현재 User-Agent 헤더를 요구하며, 고빈도 요청 시 차단될 수 있다. AGENTS.md는 "API 키나 토큰을 파일에 하드코딩하지 않는다"고 명시하며, 자동 수집 구현 시 이 금지 조항을 위반할 가능성이 있다.
- **Hood chain RPC 엔드포인트**: Hood chain 테스트넷 또는 메인넷 RPC URL과 지갑 개인키(또는 니모닉)가 코드에 노출될 위험이 있다. AGENTS.md 금지 조항과 직접 충돌한다.
- **외부 가격 API**: 실시간 시세 반영을 위해 CoinGecko, CryptoCompare 등을 사용하면 API 키가 필요할 수 있다. AGENTS.md 금지 조항에 해당하므로 이번 범위에서 완전 제외해야 한다.
- **결론**: Day7·8 범위에서 외부 네트워크 호출이 필요한 기능은 모두 제외하거나 로컬 파일·하드코딩 픽스처로 대체해야 AGENTS.md 금지 조항을 준수할 수 있다.

### 검증 누락 항목

PROJECT_QUESTIONS.md Q7 기준으로, 현재 기획자·구현자 의견에는 구체적인 검증 기준이 없다. VERIFY.md에 반드시 기록해야 하는 항목이 누락되어 있다.

- **파싱 정확도 기준 미설정**: "파싱이 된다"가 아니라 "파싱한 종목명·수량이 원본 XML의 `<nameOfIssuer>`·`<sshPrnamt>` 값과 일치하는가"처럼 필드 단위 일치 기준이 없다.
- **비중 계산 오차 허용 범위 미설정**: 부동소수점 연산 결과가 소수점 몇 자리까지 정확해야 하는지 기준이 없다.
- **localStorage 복원 검증 기준 미설정**: 새로고침 후 테이블 행 수와 데이터가 파싱 직후와 동일한지 확인하는 기준이 없다.
- **모바일 레이아웃 검증 기준 미설정**: AGENTS.md가 "텍스트가 모바일에서 겹치지 않게" 명시하나, 어떤 뷰포트 너비에서 테스트할지 기준이 없다.
